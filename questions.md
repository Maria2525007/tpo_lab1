
# Лабораторная работа №1
## Дисциплина: Тестирование программного обеспечения
## Ответы на вопросы к защите

---

### 1. Понятие тестирования ПО. Основные определения

**Тестирование программного обеспечения** — процесс проверки и анализа ПО с целью выявления дефектов и подтверждения соответствия требованиям.

**Основные термины:**

| Термин | Определение |
|--------|-------------|
| **Ошибка (error)** | Человеческая ошибка при проектировании или написании кода |
| **Дефект (bug)** | Ошибка в коде или логике системы |
| **Сбой (failure)** | Проявление дефекта при выполнении программы |
| **Требование** | Формализованное описание ожидаемого поведения системы |
| **Тест-кейс** | Набор входных данных, шагов и ожидаемого результата |
| **Оракул** | Способ определить корректность результата (спецификация, эталонная функция, математическая формула) |

---

### 2. Цели тестирования. Классификация тестов

**Цели тестирования:**

- ✅ выявление дефектов
- ✅ подтверждение соответствия требованиям
- ✅ снижение рисков
- ✅ оценка качества ПО
- ✅ предотвращение регрессий

**Классификация тестов:**

| Критерий | Виды |
|----------|------|
| **По уровню** | Модульные (Unit) / Интеграционные / Системные / Приемочные |
| **По подходу** | Функциональные / Нефункциональные (производительность, безопасность и др.) |
| **По знанию внутренней структуры** | Black-box / White-box / Gray-box |
| **По цели** | Smoke / Sanity / Regression |

---

### 3. Модульное тестирование. Понятие модуля

**Модульное тестирование (Unit testing)** — тестирование отдельного компонента системы (метода, функции, класса) изолированно от других частей.

**Модуль** — минимальная логическая единица программы, которую можно протестировать отдельно.

**Признаки хорошего unit-теста:**

- 🔹 независимость
- 🔹 детерминированность
- 🔹 быстрый запуск
- 🔹 проверка одной логической идеи
- 🔹 понятный ожидаемый результат

---

### 4. V-образная модель. Статическое и динамическое тестирование

**V-модель разработки** связывает этапы проектирования и тестирования:

```
Требования ──────────────► Приемочное тестирование
     ▲                            ▲
     │                            │
Архитектура ────────────► Интеграционное тестирование
     ▲                            ▲
     │                            │
Проектирование модуля ──► Модульное тестирование
     ▲                            ▲
     │                            │
     └────── Кодирование ─────────┘
```

**Статическое тестирование:**
- без запуска кода
- ревью, инспекции, анализ требований
- линтеры, статический анализ

**Динамическое тестирование:**
- с запуском программы
- unit, integration, system tests

### 5. Валидация и верификация. Black-box и White-box

**Верификация** — проверка соответствия продукта спецификации.
> *"Делаем ли мы продукт правильно?"*

**Валидация** — проверка соответствия ожиданиям пользователя.
> *"Тот ли продукт мы делаем?"*

**Подходы к тестированию:**

| Подход | Описание |
|--------|----------|
| **Black-box** | Тестирование по входам и выходам без анализа кода |
| **White-box** | Тестирование с учетом внутренней структуры программы (ветви, условия, пути выполнения) |
| **Gray-box** | Комбинация подходов, частичное знание внутренней структуры |

---

### 6. Тестовый случай, сценарий и тестовое покрытие

**Тестовый случай (Test Case)** — конкретный набор входных данных и ожидаемый результат.

**Тестовый сценарий** — последовательность действий, объединяющая несколько тест-кейсов.

**Тестовое покрытие** — степень охвата требований или кода тестами.

**Виды покрытия:**

- 📊 покрытие требований
- 📊 покрытие операторов
- 📊 покрытие ветвлений
- 📊 покрытие условий
- 📊 покрытие классов эквивалентности

---

### 7. Анализ эквивалентности

**Метод тест-дизайна**, при котором входные данные разбиваются на группы (эквивалентные классы), внутри которых программа должна вести себя одинаково.

**Принцип работы:**
1. Выделение классов эквивалентности
2. Выбор одного представителя из каждого класса
3. Тестирование только выбранных представителей

**Преимущество:** сокращение числа тестов без потери качества

**Пример для поля ввода возраста (0-120):**

| Класс | Диапазон | Представитель |
|-------|----------|---------------|
| Недопустимый (ниже) | < 0 | -5 |
| **Допустимый** | **0 - 120** | **25** |
| Недопустимый (выше) | > 120 | 150 |

### 8. Таблицы решений и таблицы переходов

**Таблица решений**
- Используется при сложной логике с несколькими условиями
- Позволяет формализовать комбинации условий и ожидаемые действия

**Пример таблицы решений:**

| Условие 1 | Условие 2 | Условие 3 | Действие |
|-----------|-----------|-----------|----------|
| Да | Да | — | A |
| Да | Нет | Да | B |
| Да | Нет | Нет | C |
| Нет | — | — | D |

**Таблица переходов**
- Используется, когда система имеет состояния
- Определяет переходы: **Состояние + Событие → Новое состояние**
- Применяется при моделировании сценариев и бизнес-логики

**Пример таблицы переходов:**

| Текущее состояние | Событие | Новое состояние |
|-------------------|---------|-----------------|
| Новый заказ | Оплата | Оплачен |
| Оплачен | Отправка | Отправлен |
| Отправлен | Доставка | Доставлен |
| Доставлен | Завершение | Завершен |

### 9. Регрессионное тестирование

**Регрессионное тестирование** — повторное выполнение тестов после изменений в коде для проверки, что ранее работавшая функциональность не нарушена.

**Проводится:**

- 🔄 после исправления багов
- 🔄 перед релизом
- 🔄 при добавлении нового функционала
- 🔄 при рефакторинге кода

**Важно:** автоматизация регрессионных тестов позволяет значительно ускорить процесс проверки.

### 10. JUnit 5. Особенности API. Класс junit.framework

**JUnit 5 (Jupiter)** — современный фреймворк для модульного тестирования Java.

**Основные аннотации:**

| Аннотация | Назначение |
|-----------|------------|
| `@Test` | Помечает метод как тестовый |
| `@BeforeEach` | Выполняется перед каждым тестом |
| `@AfterEach` | Выполняется после каждого теста |
| `@BeforeAll` | Выполняется один раз перед всеми тестами |
| `@AfterAll` | Выполняется один раз после всех тестов |
| `@ParameterizedTest` | Параметризованный тест |

**Основные методы проверок (assertions):**

```java
assertEquals(expected, actual);
assertTrue(condition);
assertFalse(condition);
assertThrows(Exception.class, () -> { ... });
assertAll(() -> test1(), () -> test2());
```

**Возможности JUnit 5:**

- ✅ параметризованные тесты
- ✅ теги и фильтрация
- ✅ расширяемость через extensions
- ✅ современный API
- ✅ поддержка динамических тестов

---

**junit.framework**

Пакет старой версии JUnit (JUnit 3).

| Характеристика | JUnit 3 | JUnit 5 |
|----------------|---------|---------|
| **Подход** | Наследование от `TestCase` | Аннотации |
| **Именование** | Методы должны начинаться с `test` | Любое имя с `@Test` |
| **Гибкость** | Низкая | Высокая |
| **Расширяемость** | Ограниченная | Мощная (Extensions) |

> ⚠️ **Важно:** В современных проектах практически не применяется. Рекомендуется использовать JUnit 5 (Jupiter).
